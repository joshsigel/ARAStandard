# Foreword
This document is a companion to the Autonomous Reliability Assurance (ARA) Standard, ARAF-ARA-STD-2026-002. It provides the complete technical reference for the ARA Behavioral Telemetry SDK, the embeddable software component that instruments autonomous systems for runtime monitoring, continuous assurance data collection, and integration with the Continuous Assurance Platform (CAP) operated by an accredited Continuous Assurance Platform Operator (CAPO). The ARAF ecosystem supports multiple accredited CAPOs; certified organizations select their preferred CAPO from the ARAF CAPO Directory (see ARAF-ARA-GOV-2026-001, Article 4 for CAPO accreditation requirements). The SDK’s telemetry interface is identical regardless of which accredited CAPO receives the data.
The Behavioral Telemetry SDK is not an optional monitoring add-on. It is the primary mechanism through which certified autonomous systems maintain their certification as a persistent operational state. The SDK provides the telemetry pipeline that feeds drift detection, anomaly alerting, policy compliance verification, and automated revalidation. Without this integration, certification degrades from a live signal to a stale credential.
This reference is designed for three audiences: (1) platform engineers integrating the SDK into autonomous systems seeking or maintaining ARA certification; (2) Authorized Validation Bodies (AVBs) verifying that SDK integration meets ARA monitoring requirements; and (3) security and compliance teams evaluating the SDK’s data handling, access controls, and privacy characteristics.
The SDK supports five integration methods: REST API, gRPC, and native language bindings for Python, Node.js, Java, Go, and Rust. All methods produce identical telemetry streams and are interchangeable from the platform’s perspective. Selection is a deployment decision, not a compliance decision.
Design Philosophy: The SDK treats telemetry as a first-class operational concern, not an afterthought. An autonomous system that cannot be observed cannot be trusted. The SDK makes observation continuous, structured, and tamper-evident.

# Article 1 — Scope and Purpose

## 1.1 Purpose
This document specifies the technical interface, data schemas, integration patterns, security requirements, and operational guidance for the ARA Behavioral Telemetry SDK. It establishes the normative requirements that SDK implementations SHALL satisfy to support ARA certification and continuous assurance monitoring.

## 1.2 Applicability
This reference applies to all autonomous systems seeking or maintaining ARA certification at any certification level (Level 1 Foundation, Level 2 Operational, Level 3 Comprehensive) and any system profile (Foundational, Standard, Advanced, Comprehensive). The SDK requirements apply equally to systems evaluated via the self-serve platform pathway and those evaluated through AVB engagement.
The SDK is required for systems enrolling in the Continuous Assurance Program. Systems using equivalent monitoring integrations that satisfy the data schema, delivery, integrity, and security requirements specified in this document may petition the ARAF Technical Standards Board for equivalence determination.
Platform Certification Telemetry Exception. Platform Certifications evaluated within a Reference Environment as defined in ARAF-ARA-EVL-2026-002 Article 3.8.3 may use simulated telemetry in lieu of live operational telemetry for the duration of the evaluation. The simulated telemetry mechanism SHALL be documented in the Reference Environment Specification (RES) and attested by the evaluating AVB or platform operator per ACR-1.26. Certified Platform Certifications that are subsequently integrated into deployer systems do not transfer this exception; deployer systems remain subject to all SDK telemetry requirements applicable to their Assurance Class and certification level.

# Article 2 — Architecture Overview

## 2.1 System Context
The Behavioral Telemetry SDK operates as an embedded component within the autonomous system’s runtime environment. It intercepts, structures, and transmits telemetry data to the accredited CAPO’s CAP for ingestion, analysis, and compliance evaluation. The SDK is positioned between the autonomous system’s operational logic and the ARA Continuous Assurance Platform.
Agent-Side Components: The SDK library embedded in the autonomous system’s process space. Responsible for telemetry capture, local buffering, schema validation, and secure transmission.
Platform-Side Components: The accredited CAPO’s platform telemetry ingestion endpoints, stream processors, and storage layer. Responsible for receiving, validating, storing, and routing telemetry to downstream analysis engines (Drift Detection, Scoring, Alerting). CAPOs are independently accredited by ARAF to operate this infrastructure. The SDK transmits telemetry to the certified organization’s selected CAPO; all accredited CAPOs implement the same ingestion interface specification defined in this document.

## 2.3 Data Flow Model
Telemetry flows through the following stages:
1. Capture: The SDK instruments the autonomous system to capture events at defined instrumentation points (action execution, decision evaluation, boundary checks, resource consumption). Events are captured with nanosecond-precision timestamps.
2. Structuring: Captured events are structured into one of the seven canonical schemas (Action Trace, Decision Log, Boundary Event, Drift Signal, Resource Utilization, Privacy Event, Societal Impact Event) defined in Article 4.
3. Validation: The SDK validates each event against its schema before transmission, rejecting malformed events locally and logging validation failures.
4. Privacy Filtering: For systems with privacy-preserving telemetry modes enabled (see Article 7), the SDK applies PII detection, data minimization, and differential privacy transformations before buffering.
5. Buffering: Events are buffered locally in a persistent queue to survive transient network failures. The buffer is configurable (see Article 8) with default capacity for 24 hours of telemetry at peak volume.
6. Transmission: Buffered events are transmitted to the accredited CAPO’s CAP via the configured integration method (REST, gRPC, or native binding transport). All transmissions use TLS 1.3 encryption.
7. Acknowledgment: The platform acknowledges receipt of each batch. Unacknowledged batches are retried with exponential backoff. Events are removed from the local buffer only after platform acknowledgment.

## 2.4 Security Architecture
The SDK’s security model operates on five principles:
Zero Trust Transmission: All data in transit is encrypted with TLS 1.3. Mutual TLS (mTLS) is available and REQUIRED for Level 3 certifications.
Tamper-Evident Storage: All telemetry data is hashed using SHA-256 at capture time. Hash chains link sequential events, making undetected modification computationally infeasible.
Least-Privilege Access: SDK credentials grant write-only access to telemetry ingestion endpoints. The SDK cannot read, modify, or delete previously submitted telemetry.
Credential Isolation: SDK authentication credentials are scoped to a single registered system and cannot be used to access telemetry from other systems.
Privacy by Design: The SDK implements data minimization, PII detection, and configurable redaction at the point of capture, ensuring sensitive data is never transmitted when privacy-preserving modes are enabled.

## 3.2 gRPC Interface
The gRPC interface provides high-throughput, low-latency streaming suitable for systems generating large volumes of telemetry. It supports both unary and bidirectional streaming RPCs.

### Service Definition
The ARAtelemetry gRPC service includes unary RPCs for batched submission of each event type (SubmitTraces, SubmitDecisions, SubmitBoundaries, SubmitDrift, SubmitResources, SubmitPrivacy, SubmitSocietal, CheckIn), client streaming RPCs for continuous telemetry (StreamTraces, StreamAll), and bidirectional streaming (Monitor) for simultaneous telemetry and platform directives.

### Connection Management
gRPC connections SHALL use TLS 1.3 with ALPN negotiation for HTTP/2. Clients SHALL implement automatic reconnection with exponential backoff (initial delay 100ms, maximum delay 30s, jitter factor 0.2). Channel credentials are provided via the same JWT mechanism used by the REST API, transmitted as gRPC metadata on each call.

### Streaming Modes
The bidirectional Monitor RPC is RECOMMENDED for Level 2 Operational and REQUIRED for Level 3 Comprehensive certifications, as it enables the platform to issue revalidation triggers and configuration directives in real time.

## 3.3 Native Language Bindings
The SDK provides idiomatic client libraries for five languages. Each binding wraps the gRPC or REST transport layer and provides language-native APIs for telemetry capture.

## 3.4 Agent-to-Agent Handshake Protocol
For autonomous AI agents capable of programmatic interaction, the SDK supports an agent-to-agent handshake protocol. This enables AI agents to autonomously initiate SDK integration, negotiate telemetry parameters, and begin streaming telemetry without human configuration.

### Handshake Sequence
1. Discovery: The agent queries the accredited CAPO’s CAP discovery endpoint (/.well-known/ara-telemetry) to obtain connection parameters, supported schemas, and authentication requirements.
2. Registration: The agent presents its system registration credentials and requests telemetry channel provisioning.
3. Capability Negotiation: The platform and agent negotiate telemetry parameters: schema version, transport protocol, compression, batch size, and streaming mode.
4. Channel Establishment: A dedicated telemetry channel is provisioned with the negotiated parameters. The agent receives a channel token scoped to this session.
5. Continuous Operation: The agent streams telemetry on the established channel. The platform streams directives (revalidation triggers, schema updates) back to the agent.
This protocol is specifically designed for the ‘agents certifying agents’ paradigm described in the ARA Standard. It enables certified AI agents to maintain their own certification by autonomously managing their telemetry integration.

### 3.4.1 AAIF/MCP Protocol Interoperability
Behavioral telemetry collected via the ARA SDK is transport-agnostic and is designed to operate across AAIF-governed protocols, including systems implementing the Model Context Protocol (MCP). The Agentic AI Foundation (AAIF), a Linux Foundation initiative (founded December 2025), governs open interoperability standards for agentic AI systems at the connectivity layer. ARA SDK telemetry operates at the behavioral reliability layer; MCP operates at the connectivity and tool invocation layer. These layers are complementary and are designed to coexist without conflict. MCP-compliant deployments are encouraged to surface ARA telemetry endpoints within their tool registration schemas to enable continuous reliability monitoring alongside protocol-level interoperability. AAIF compliance does not reduce, replace, or substitute for ARA SDK telemetry requirements.

# Article 4 — Data Schemas
The SDK defines seven canonical telemetry schemas. All telemetry submitted to an accredited CAPO’s CAP SHALL conform to one of these schemas. Schemas are versioned independently and the platform supports concurrent schema versions during migration periods.

## 4.1 Action Trace Schema
Action traces record individual actions taken by the autonomous system. Every autonomous action that affects external state, consumes resources, or exercises delegated authority SHALL be captured as an action trace.
ACR Alignment: The action trace schema directly supports ACR-10.01 (action logging with type, timestamp, input context, rationale, and outcome), ACR-10.05 (tamper-evident logging via integrity_hash and chain_hash), and ACR-12.01 (decision log maintenance).

## 4.2 Decision Log Schema
Decision logs capture the decision-making process for autonomous decisions that select between alternatives, apply judgment, or exercise discretion. While action traces record what happened, decision logs record why.

## 4.3 Boundary Event Schema
Boundary events record when the autonomous system approaches, reaches, or violates its certified operational boundaries. These events are critical for drift detection and policy compliance monitoring.

## 4.4 Drift Signal Schema
Drift signals report changes in the autonomous system’s behavioral distribution relative to its certified baseline. The SDK’s embedded anomaly detection generates drift signals; the platform’s Drift Detection Engine performs deeper analysis.

## 4.5 Resource Utilization Schema
Resource utilization events track the autonomous system’s consumption of computational, API, and operational resources. These events support capacity planning, cost monitoring, and detection of resource-based anomalies.

## 4.6 Privacy Event Schema (Domain 5)
Privacy events capture data handling activities relevant to Domain 5 (Data Privacy and Information Protection). These events enable continuous monitoring of PII handling, consent management, cross-tenant isolation, and data breach detection. The privacy event schema provides the telemetry foundation for demonstrating compliance with ACRs 5.01 through 5.24.
ACR Alignment: The privacy event schema directly supports ACR-5.09 (PII detection and prevention), ACR-5.06 (consent management), ACR-5.10 (cross-tenant isolation), ACR-5.12 (telemetry sanitization), ACR-5.24 (breach detection), and ACR-5.04 (data flow mapping).

## 4.7 Societal Impact Event Schema (Domain 13)
Societal impact events capture activities relevant to Domain 13 (Societal Impact and Responsible Deployment). These events enable continuous monitoring of content safety guardrails, bias and fairness metrics, CBRN/cyber safety controls, and hallucination detection. The societal impact event schema provides the telemetry foundation for demonstrating compliance with ACRs 13.01 through 13.22.
ACR Alignment: The societal impact event schema directly supports ACR-13.01 (CBRN guardrails), ACR-13.02 (cyber guardrails), ACR-13.04 (harmful content safeguards), ACR-13.06 (hallucination detection), ACR-13.09 (hate speech controls), ACR-13.10 (bias monitoring), and ACR-13.08 (deepfake detection).

## 4.8 Schema Versioning
All schemas follow semantic versioning (MAJOR.MINOR). Major version changes indicate backward-incompatible changes requiring SDK upgrades. Minor version changes add optional fields and are backward-compatible. The platform SHALL support at least the two most recent major versions concurrently during migration periods (minimum 6 months overlap).
Schema versions are declared per-event in the schema_version field. The SDK SHALL reject events with unrecognized schema versions and log validation errors locally.

# Article 6 — Continuous Assurance Platform Integration

## 6.1 Internal Integration
Internal integration connects the SDK directly to the autonomous system’s runtime environment. This is the standard integration pattern for single-organization deployments.
Connection Lifecycle: The SDK establishes a persistent connection to the accredited CAPO’s CAP upon system startup. This connection is maintained throughout the system’s operational lifetime. If the connection is interrupted, the SDK buffers events locally and reconnects automatically. The maximum buffer duration is configurable (default: 24 hours). Events exceeding the buffer duration are persisted to local storage with integrity hashes for later reconciliation.
Telemetry Channels: Internal integration uses dedicated telemetry channels per event type (traces, decisions, boundaries, drift, resources, privacy, societal). Channels operate independently; failure on one channel does not affect others. Each channel maintains its own buffer, retry state, and acknowledgment tracking.
Platform Directives: The accredited CAPO’s CAP sends directives to the SDK via a reverse channel (gRPC server-streaming or WebSocket for REST). Directives include revalidation triggers, configuration update requests, schema migration notices, and emergency halt commands. The SDK SHALL process all directives within 30 seconds of receipt.

## 6.2 Multi-Party Integration
Multi-party integration enables monitoring of autonomous systems operating across organizational boundaries. This is the integration pattern for settlement platforms, multi-agent orchestration, supply chain coordination, and cross-company exception resolution.
In multi-party mode, each organization deploys its own SDK instance connected to its autonomous system. The accredited CAPO’s CAP provides a shared assurance layer that monitors the interaction boundary between parties without exposing proprietary operational data.
Data Isolation: Each party’s internal telemetry is visible only to that party and the accredited CAPO’s compliance engine. No party can access another party’s raw telemetry.
Interaction Boundary Monitoring: The accredited CAPO’s CAP monitors the shared interaction surface (API calls between parties, shared data exchanges, coordinated decisions) and generates cross-party compliance signals.
Verified Compliance Signals: Each party receives verified compliance signals about their counterparties’ certification status, boundary adherence, and behavioral reliability. These signals are cryptographically signed by the accredited CAPO.

## 6.3 Telemetry Ingestion Pipeline
The accredited CAPO’s CAP ingestion pipeline processes telemetry through the following stages:
1. Reception: Events arrive via REST or gRPC endpoints. Load balancers distribute across ingestion nodes. Each event is assigned a platform-side receipt timestamp.
2. Validation: Schema validation, integrity hash verification, and duplicate detection. Invalid events are rejected with diagnostic error codes.
3. Enrichment: Events are enriched with platform metadata: system certification context, applicable ACR mappings, domain classification, risk weight, and enterprise risk dimension assignment.
4. Routing: Enriched events are routed to downstream consumers: Drift Detection Engine, Scoring Engine, Alerting System, Insurance Data Feed, and archival storage. Routing is event-type-aware.
5. Storage: Events are stored in tamper-evident, append-only storage with retention periods aligned to CAPO SLA requirements (ARAF-ARA-GOV-2026-001, Article 4.5): minimum 13 months for telemetry data (compliance evidence), minimum 7 years for audit trail and certification-relevant events. The SDK buffers locally for up to the configured duration before reconciliation with the accredited CAPO’s storage layer.

## 6.4 Real-Time Alerting
The accredited CAPO’s CAP generates real-time alerts based on telemetry analysis. Alerts are classified by severity and routed to configured notification channels.

## 6.5 Compliance Dashboard Data Feeds
The SDK and CAP together provide data feeds for the real-time compliance dashboard. The dashboard is accessible to system operators, management, AVBs, and auditors (with role-based access controls).

## 6.6 Insurance Data Feed Specification
The insurance data feed provides a standardized interface for sharing real-time certification health signals with Recognized Insurer Partners (RIPs). RIPs are insurance companies formally recognized by ARAF under ARAF-ARA-GOV-2026-001, Article 5. Only accredited CAPOs may provision insurance data feed access; insurers connect to the feed through a CAPO that has completed ARAF integration testing with that insurer (GOV Article 5.3). This enables insurance providers to incorporate autonomous system reliability data into underwriting, continuous monitoring, and claims processing. Full integration specifications will be defined in ARAF-ARA-INS-2026-001 (forthcoming).

## 7.3 Encryption
In Transit: All telemetry data SHALL be encrypted using TLS 1.3. The SDK SHALL reject connections that negotiate lower TLS versions. For gRPC, ALPN negotiation for HTTP/2 over TLS is mandatory. Cipher suites are restricted to AEAD constructions (AES-256-GCM, ChaCha20-Poly1305).
At Rest: Telemetry stored in the platform’s archival layer is encrypted using AES-256-GCM with keys managed through the platform’s key management service. Encryption keys are rotated quarterly (minimum) and on-demand after security incidents.
Local Buffer: The SDK’s local event buffer SHALL be encrypted at rest using a key derived from the system’s registration credentials. Buffer encryption uses AES-256-GCM. If the system terminates abnormally, the encrypted buffer persists and is reconciled on restart.

## 7.4 Telemetry Data Integrity
Every telemetry event includes two integrity fields:
integrity_hash: SHA-256 hash computed over the event payload (excluding the integrity_hash and chain_hash fields themselves). This enables detection of individual event tampering.
chain_hash: SHA-256 hash computed over the concatenation of the current event’s integrity_hash and the previous event’s chain_hash. This creates a sequential hash chain that makes insertion, deletion, or reordering of events detectable.
The platform independently verifies both hashes upon ingestion. Hash verification failures are logged as security events and trigger immediate investigation. Systems with persistent hash verification failures will have their certification suspended.
Data Redaction: Certain telemetry fields may contain sensitive information (PII, credentials, proprietary business logic). The SDK provides configurable redaction rules that hash or mask sensitive field values before transmission. Redaction is applied at capture time and is irreversible. Redaction rules SHALL be documented in the system’s certification evidence.

## 7.5 Privacy-Preserving Telemetry Modes
The SDK provides three privacy-preserving telemetry modes that enable organizations to maintain ARA compliance while meeting stringent data privacy requirements. These modes are configured per-system and can be combined. Accredited CAPOs are required to support all three privacy-preserving modes defined in this Article (ARAF-ARA-GOV-2026-001, Article 4.2). Organizations selecting a CAPO should verify that the CAPO’s platform supports the privacy modes required for their deployment context before finalizing their CAPO selection.

### 7.5.1 Data Minimization Mode
When enabled, the SDK automatically strips all optional fields from telemetry events, transmitting only the minimum required fields per schema. Input and output context objects are replaced with structural summaries (field counts, data types, size) rather than actual content. This mode reduces the risk of inadvertent sensitive data transmission while maintaining compliance with all mandatory ACR monitoring requirements.

### 7.5.2 Local-Only Processing Mode
Local-only processing mode enables the SDK to perform privacy-sensitive analysis (PII detection, content classification, bias evaluation) entirely within the autonomous system’s local environment. Only aggregate results and compliance signals are transmitted to the platform. Raw data never leaves the local environment.
This mode is designed for systems processing highly sensitive data (healthcare, financial, government) where data residency requirements prohibit transmission of raw operational data to external platforms. The SDK’s local analysis engine performs PII detection using on-device NER models, content safety classification, and basic drift detection locally. Compliance signals derived from local analysis are cryptographically signed and transmitted as attestations rather than raw evidence.

### 7.5.3 Differential Privacy Mode
Differential privacy mode applies calibrated noise to aggregate telemetry metrics before transmission, providing formal privacy guarantees while maintaining statistical utility for compliance monitoring.
Differential privacy mode is RECOMMENDED for systems that report aggregate metrics (drift signals, resource utilization trends, fairness metrics) where individual event privacy is a concern. It is NOT suitable for event-level telemetry required for Level 3 Comprehensive certifications.

## 7.7 Audit Logging
The SDK SHALL maintain a local audit log of all SDK operations independent of the telemetry stream. This audit log captures: SDK startup/shutdown, connection establishment and disconnection, authentication events, transmission failures and retries, configuration changes, schema validation failures, privacy mode changes, and security events. ## 7.8 Assurance Class Telemetry Obligations
The ARA three-axis certification model assigns each certified system to an Assurance Class that determines the intensity and continuity of runtime telemetry obligations. SDK integration requirements vary by Assurance Class as follows.
Class A — Periodic. Continuous real-time telemetry is not required for Class A systems. SDK integration is optional but strongly recommended as the primary mechanism for automating periodic compliance reporting and as the preferred pathway for systems that may seek to upgrade to a higher Assurance Class in the future. Class A systems that deploy the SDK MAY operate in periodic or event-triggered check-in mode per section 5.3.
Class B — Monitored. Real-time telemetry transmission is required for Class B systems. SDK integration is mandatory. Class B systems SHALL emit drift detection events per the DriftSignal schema (section 4.5) and SHALL operate in event-triggered or continuous check-in mode. The SDK SHALL be configured to transmit telemetry to an accredited CAPO’s CAP without requiring manual batch submission.
Class C — Continuously Assured. Full SDK integration is mandatory for Class C systems. All seven telemetry event types defined in Article 4 are required. Class C systems SHALL operate in continuous check-in mode and SHALL emit anomaly events within the latency thresholds specified in the applicable CAPO SLA. Drift detection, boundary status, and all Domain 5 and Domain 13 telemetry hooks SHALL be active. SDK configuration SHALL enforce mTLS per section 7.1 and SHALL not disable any telemetry channel without an approved variance on file with ARAF.

# Article 9 — Multi-Party Integration Specifications

## 9.1 Cross-Organization Architecture
Multi-party integration addresses the assurance challenges of autonomous systems that operate across organizational boundaries. In these deployments, reliability cannot be established by monitoring a single system in isolation; the interaction surface between parties creates its own reliability risks.
The ARA multi-party architecture defines three roles:
Participating Party: An organization operating a certified autonomous system that interacts with other parties’ systems. Each party deploys its own SDK instance and maintains its own certification.
Interaction Boundary: The defined interface between parties’ systems (APIs, data exchanges, shared resources, coordinated decisions). The boundary is the object of cross-party monitoring.
Neutral Assurance Layer: The ARA CAP operating as a trusted intermediary that monitors the interaction boundary and provides each party with verified compliance signals about their counterparties.

## 9.2 Party Registration and Identity
Each party in a multi-party deployment SHALL register independently with their accredited CAPO’s CAP. Party registration includes: organization identity verification, system registration for each participating autonomous system, interaction boundary definition (jointly declared by all parties), and data sharing agreements specifying what compliance signals are shared with counterparties.
Party identities are cryptographically bound to their SDK instances. Cross-party compliance signals include the issuing party’s verified identity, preventing impersonation or spoofing of compliance status.

## 9.5 Data Isolation and Privacy
Multi-party data isolation is enforced at the platform level through cryptographic access controls. Each party’s telemetry is encrypted with party-specific keys. The accredited CAPO’s CAP compliance engine processes telemetry in isolated execution contexts. Cross-party compliance signals are generated from aggregate analysis, not raw data access. Audit logs record all cross-party signal generation and delivery.
The multi-party architecture is designed to provide assurance without trust. Each party receives verified signals from a neutral platform without requiring access to, or trust in, the other party’s internal operations.

## 10.2 High Availability
The SDK is designed to operate reliably even when the platform is temporarily unavailable. High availability mechanisms include:
Local Buffering: Events are buffered locally during platform outages. The buffer survives process restarts (disk-backed) and SDK upgrades. Buffer integrity is maintained via the same hash chain used for telemetry events.
Graceful Degradation: The SDK SHALL NOT cause the autonomous system to fail if the platform is unreachable. Telemetry capture continues locally. The SDK reports its connectivity status via the metrics endpoint.
Reconciliation: When connectivity is restored, the SDK transmits buffered events in chronological order. The platform reconciles buffered events with its timeline and recomputes compliance scores retroactively.

## 10.4 Versioning and Upgrades
SDK versions follow semantic versioning (MAJOR.MINOR.PATCH). The platform guarantees backward compatibility within a major version.
Patch Updates (x.x.PATCH): Bug fixes only. Drop-in replacement with no configuration changes. Zero downtime upgrade supported.
Minor Updates (x.MINOR.x): New features and optional schema fields. Drop-in replacement. New features require explicit opt-in via configuration.
Major Updates (MAJOR.x.x): Potentially breaking changes. Migration guide provided. Minimum 6-month overlap period where both old and new major versions are supported. Revalidation of monitoring-related ACRs (Domain 10) is required after major version upgrade.

Term | Definition
ACR | Autonomous Control Requirement. Individual testable requirement within the ARA Standard.
Action Trace | Telemetry record of a single autonomous action including inputs, outputs, timing, and outcome.
ARAF | Autonomous Reliability Assurance Foundation. The independent standards body governing ARA certification.
ARA | Autonomous Reliability Assurance. The certification standard published by ARAF.
AVB | Authorized Validation Body. Third-party organization authorized by ARAF to conduct ARA evaluations.
Boundary Event | Telemetry record of an operational boundary approach, reach, violation, or recovery.
CAP | Continuous Assurance Platform. The technology infrastructure operated by an accredited CAPO for real-time telemetry ingestion and compliance monitoring. The SDK transmits to the CAP; CAPOs are the entities that operate and maintain it.
CAPO | Continuous Assurance Platform Operator. An ARAF-accredited company that operates a CAP and provides telemetry ingestion, compliance monitoring, insurance data feeds, and ARAF Interoperability Protocol support. Certified organizations select their CAPO from the ARAF CAPO Directory. See ARAF-ARA-GOV-2026-001, Article 4.
Chain Hash | Sequential hash linking consecutive telemetry events to detect insertion, deletion, or reordering.
Check-In | Structured assertion of operational state submitted by an autonomous system to its accredited CAPO’s CAP.
Decision Log | Telemetry record of an autonomous decision including alternatives, rationale, and confidence.
Differential Privacy | Mathematical framework providing formal privacy guarantees by adding calibrated noise to aggregate data.
Drift Signal | Telemetry record indicating behavioral deviation from certified baseline.
Enterprise Risk Dimension | One of six organizational risk categories grouping related ARA domains for governance reporting.
HITL | Human-in-the-Loop. Human participation in governance, review, and certification decision-making.
Integrity Hash | SHA-256 hash of a telemetry event payload for individual tamper detection.
Insurance Data Feed | Standardized API providing risk-relevant certification signals to partner insurance providers.
mTLS | Mutual TLS. Two-way certificate authentication required for Level 3 certifications.
Multi-Party Integration | CAP capability for monitoring autonomous systems operating across organizational boundaries.
PII Detection | Automated identification of personally identifiable information in telemetry data streams.
Privacy Event | Telemetry record of data privacy activities including PII detection, consent, and breach signals.
Revalidation | Re-evaluation of specific ACRs triggered by material system changes.
SDK | Software Development Kit. The Behavioral Telemetry SDK documented in this reference.
Societal Impact Event | Telemetry record of content safety, bias, and responsible deployment control activities.
Statement of Applicability | Organization’s documented justification for inclusion/exclusion of each ACR, modeled on ISO 27001.
System Profile | Classification of system complexity determining applicable ACRs: Foundational, Standard, Advanced, Comprehensive.
TSB | Technical Standards Board. Ultimate technical authority within ARAF.
Assurance Class | Classification assigned to a certified system that determines the intensity of runtime monitoring obligations. Three classes: Class A (Periodic), Class B (Monitored), Class C (Continuously Assured).
Assurance Lapse | The condition in which a certified system fails to meet the continuous telemetry or check-in obligations required for its Assurance Class, resulting in automatic downgrade of certification status.
Certification Inheritance | The conditional transfer of a platform vendor’s Platform Certification to deployer systems that integrate the certified platform component, subject to Deployment Certification and compliance with all applicable ACRs not covered by the platform scope.
Certified Technology Partner (CTP) | A consulting or implementation firm listed in the ARAF CTP Registry as having demonstrated ARA implementation competency. Not a certification — a registry designation. Does not confer evaluation authority or certification rights.
Class A — Periodic | Assurance Class designation for certified systems with lower operational risk profiles. Requires periodic compliance reporting. Continuous real-time telemetry is not mandated; SDK integration is optional.
Class B — Monitored | Assurance Class designation for certified systems with elevated operational risk profiles. Requires real-time telemetry transmission and mandatory SDK integration. Drift detection events are required.
Class C — Continuously Assured | Assurance Class designation for certified systems with the highest operational risk profiles. Requires full SDK integration, all seven telemetry event types, continuous check-in mode, and anomaly event emission within CAPO SLA latency thresholds.
Deployment Certification | The ARA certification issued to an organization that deploys an autonomous system in production, covering the full deployed system including any integrated Platform Certified components. Distinct from Platform Certification.
Evaluation Level | One axis of the ARA three-axis certification model. Defines the rigor of the certification assessment: Level 1 (Foundation), Level 2 (Operational), Level 3 (Comprehensive).
Platform Certification | An ARA certification variant issued to a platform vendor or technology provider for a defined component or capability evaluated within a Reference Environment. Platform Certification covers only the ACR domains within its documented scope; deployer systems must obtain Deployment Certification for remaining ACRs.
Reference Environment | A controlled, reproducible environment specification used for evaluating Platform Certifications. Documents the simulated telemetry mechanism, system version, and ACR domain scope applicable to the evaluation.
Risk Classification | The ARA framework designation assigned to a system or component that determines its required Assurance Class and applicable ACR set, based on assessed operational risk, deployment context, and potential for harm.
